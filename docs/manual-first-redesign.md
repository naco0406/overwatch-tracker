# Manual-first redesign plan

작성일: 2026-06-01

## Product strategy

제품의 기본 입력은 스크린샷 자동 인식이 아니라 빠른 수기 기록이다. OCR은 삭제하지 않고, 스크린샷을 보조로 쓰고 싶을 때만 접근하는 고급 입력으로 둔다.

새 메인 플로우의 성공 기준은 다음과 같다.

- 홈에서 모달 없이 한 판을 저장한다.
- 필수 입력은 맵, 스코어, 결과뿐이다.
- 플레이 시간, 세션, 계정, 큐, 입력 출처는 자동으로 채운다.
- 모드는 맵 선택으로 자동 결정한다.
- 영웅, 태그, 메모, 팀 조합은 선택 정보다.
- 모바일에서도 한 손으로 맵 선택, 스코어 입력, 결과 선택, 저장이 가능해야 한다.

## UX architecture

### Home

홈은 세 영역으로 구성한다.

1. 빠른 기록
   - 맵 선택이 첫 번째 액션이다.
   - 맵 버튼은 모드별로 묶되, 작은 화면에서는 가로 스크롤/조밀한 그리드로 접근한다.
   - 스코어는 `우리 : 상대` 두 숫자 입력으로 유지한다.
   - 결과는 승리/패배/무승부 segmented control로 둔다.
   - 저장 후 같은 화면에서 폼을 초기화하고 오늘 기록 목록을 갱신한다.

2. 오늘 상태
   - 오늘 경기 수, 승률, 현재 흐름을 그대로 보여준다.
   - 최근 저장 경기와 오늘 세션 목록은 입력 직후 확인을 위해 홈에 남긴다.

3. 보조 입력
   - 이미지 붙여넣기/파일 선택/분석은 홈에서 접근 가능하지만 메인 영역을 차지하지 않는다.
   - 상세 입력 모달은 계정, 큐, 시간, 영웅, 메모, 태그까지 수정해야 할 때 사용한다.

### Records

스프레드시트 이전 관점의 전체 기록 페이지를 새로 둔다.

- 최신순 테이블을 기본으로 한다.
- 필터는 검색, 기간, 모드, 결과, 계정 정도만 둔다.
- 데스크톱은 조밀한 테이블, 모바일은 행 카드로 전환한다.
- 선택된 경기에는 벌크 액션을 제공한다.
- 벌크 수정의 1차 범위는 결과, 맵(모드 자동 동기화), 계정이다.
- 벌크 삭제는 확인 다이얼로그를 반드시 거친다.

## Data contract

기존 DB 스키마는 유지한다. 빠른 입력이 저장할 필드는 다음과 같다.

```ts
{
  playedAt: new Date().toISOString(),
  mapId,
  modeId: selectedMap.modeId,
  result,
  teamScore,
  enemyScore,
  accountId: default main account or null,
  account: accountId ? account.isMain ? 'main' : 'sub' : 'main',
  queueType: userSettings.defaultQueueType ?? 'solo',
  myHeroes: [],
  memo: '',
  tags: [],
  source: 'manual'
}
```

영웅이 비어 있어도 유효한 기록이다. 통계의 1차 축이 모드/맵/결과이기 때문에 영웅 입력은 저장 조건에서 제외한다.

## Implementation plan

1. `QuickMatchEntry` 컴포넌트를 추가한다.
   - `react-hook-form`까지 쓰지 않고 로컬 상태와 명시적 검증으로 유지한다.
   - 맵 선택은 `mapOptions`를 모드별로 묶어서 렌더링한다.
   - 저장 성공 시 폼을 맵만 유지할지 전체 초기화할지 선택할 수 있지만, 초기 구현은 스코어/결과만 초기화한다.

2. `HomePage`를 재배치한다.
   - 퀵 입력을 첫 번째 패널로 올린다.
   - 기존 OCR 드롭존은 작은 보조 패널로 축소한다.
   - 기존 `MatchEntryDialog`는 상세 입력/이미지 분석 결과 확인/수정에 계속 사용한다.

3. 상세 입력 폼의 영웅 필수 검증을 제거한다.
   - 기존 OCR/상세 입력에서도 영웅은 선택 정보가 된다.
   - “영웅 미지정” 표현을 빈 문자열 대신 명시적으로 보여준다.

4. `RecordsPage`를 추가한다.
   - `useMatches`, `useUpdateMatch`, `useDeleteMatch` 기반으로 구현한다.
   - 필터링은 클라이언트에서 처리한다.
   - 선택 상태는 페이지 내부 상태로 유지한다.
   - 벌크 액션은 선택된 id 배열에 대해 순차 실행하고, 실패 시 토스트로 알려준다.

5. 내비게이션을 조정한다.
   - 홈, 기록, 세션, 통계, 설정 5개를 모바일 하단 내비게이션에 맞춘다.
   - 마스터 데이터는 설정 또는 직접 URL로 접근 가능한 보조 화면으로 낮춘다.

## Quality checklist

- 홈에서 저장까지 모달이 열리지 않는다.
- 모바일 320px 폭에서 버튼 텍스트가 넘치지 않는다.
- 테이블은 데스크톱에서만 테이블로 보이고, 모바일은 카드형 목록으로 보인다.
- OCR 관련 파일과 기능은 삭제하지 않는다.
- `npm run lint`와 `npm run build`가 통과해야 한다.
